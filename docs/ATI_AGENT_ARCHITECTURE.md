# Архитектура ATI-агента ТехноЛогистики

## 1. Назначение

ATI-агент — отдельный сервис для автоматизации работы с MAX Messenger, ATI.su, Gmail и Google Sheets.

Главная цель: принимать заявки, готовить черновики публикаций на ATI.su, искать перевозчиков, собирать ставки и вести рыночную аналитику.

## 2. Принципы

- Безопасность важнее автономности.
- На MVP все критические действия требуют ручного подтверждения.
- Агент не трогает боевого Аюба, сайт, webhook-и, `.env` и `systemd`.
- Реальные токены и ключи не хранятся в репозитории.
- По умолчанию включён режим `DRY_RUN=true`.

## 3. Общая логика

1. MAX Messenger отправляет сообщение с ключевым словом `ЗАЯВКА`.
2. MAX Handler получает сообщение через webhook.
3. Core Orchestrator парсит текст заявки.
4. Заявка сохраняется в Google Sheets или базе.
5. Draft Builder формирует черновик публикации ATI.
6. Человек подтверждает или отклоняет действие.
7. Только после подтверждения будущий ATI Handler сможет выполнить публикацию или отправку сообщения.
8. Все действия пишутся в журнал событий.

## 4. Компоненты

### MAX Handler

Принимает сообщения из MAX-группы, проверяет наличие слова `ЗАЯВКА`, передаёт текст в Orchestrator.

### Core Orchestrator

Центральная логика. Управляет парсингом, черновиками, уведомлениями, сохранением данных и будущими подтверждениями.

### ATI Handler

Будущий клиент ATI.su API. На MVP работает как заглушка и ничего не публикует при `DRY_RUN=true`.

### Gmail Handler

Будущий канал уведомлений и черновиков писем. На MVP не отправляет письма без подтверждения.

### Google Sheets Handler

Будущее хранилище MVP для заявок, перевозчиков, ставок и переговоров.

### Market Rate Analyzer

Будущий модуль анализа ставок: минимум, средняя, медиана, максимум, свежесть ответа, рекомендуемая цена клиенту.

## 5. Данные

Минимальные сущности:

- source_requests — исходные заявки;
- ati_posts — черновики и будущие публикации ATI;
- carriers — перевозчики;
- carrier_contacts — контакты;
- routes — маршруты;
- rates — ставки;
- negotiations — переговоры;
- messages — сообщения;
- approvals — подтверждения;
- audit_log — журнал действий.

## 6. MVP

Первый MVP не делает боевых действий. Он берёт тестовую заявку, парсит её, формирует черновик ATI-публикации и печатает результат в консоль.
